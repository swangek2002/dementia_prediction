问题 1：Longitudinal EHR 是怎么"塞进" transformer 的？
先澄清一个误解
你说"一个患者可能在三个集合中有记录"——其实不会。practice_id_splits 是按practice（诊所）划分的，而一个 patient 属于一个 practice（static_table 里的 PRACTICE_ID）。所以同一个病人永远只会出现在一个 split 里，不会跨 split。第二问我再展开。

一个病人 → 一个 token 序列
对一个病人，pipeline 做的事大致是：

读出该病人所有事件：从 diagnosis_table 拉出所有 (EVENT, DATE)、从 measurement_table/scripts/death 等拉出其他事件，按 DATE 排序，得到一条纯时间线：


(t1, read_code_A) → (t2, read_code_B) → (t3, bp_measurement=140) → (t4, read_code_C) → ... → (tN, last_event)
用 index_inclusion_method 做截断：根据你在 build 脚本里设的 INDEX_ON_AGE=72、study_period=[1998, 2022-10-31]、outcomes=31 dementia codes + DEATH 等规则，只保留 index_date 之前的事件作为输入，然后附加上 index_date 之后的第一个 outcome 事件（如果有的话）作为最后一个 token。这就决定了 k=1/k=2/k=0：

最后一个 token 是痴呆 code → k=1
最后一个 token 是 DEATH → k=2
index 之后没发生任何 outcome（追踪到研究期末仍未发生）→ k=0（censored），这时最后 token 是 index 之前的最后一个 GP/HES 事件
Tokenize：每个 EVENT 字符串（比如 "Eu00.", "F110.", "DEATH"，以及 measurement 分桶 token）用 meta_information_custom.pickle 里的词表映射成 integer token id。这是你说的"9万多个 token"——其实就是那 ~90k 个 Read v2 / measurement / DEATH 等词表条目，每个对应 384 维的 embedding row。

对齐成矩阵：因为不同病人序列长度不同，framework 会对一个 batch 内按最长序列 pad 到统一长度 L，再截断到 block_size=512（超过 512 会被截断，你可以在 YAML 里看到这个参数）。最后喂进模型的是：


tokens    : LongTensor [B, L]       # 每个位置是 token id
delta_t   : FloatTensor [B, L]      # 相邻事件的时间间隔（或绝对时间编码）
values    : FloatTensor [B, L]      # 连续值（比如 measurement），非连续 token 置 0
static    : FloatTensor [B, 27]     # 静态 covariate（性别、族裔、IMD、birth year…）
attn_mask : BoolTensor  [B, L]      # padding 的位置
Transformer 怎么处理它
Token embedding：[B, L] → [B, L, 384]（这就是你说的 "9万多 × 384" 的矩阵——那其实是 nn.Embedding 的权重 shape，不是单次输入的 shape）
时间编码：相邻事件的 delta_t 通过一个 time-encoding / positional-encoding 模块加到 embedding 上。这就是"longitudinal 的体现"——transformer 不需要固定的网格，它靠 delta_t 知道两个事件相隔多久
6 层 Neo block、6 head、global attention（你的 YAML 里能看到），输出 [B, L, 384]
取最后一个有效 token 的 hidden state [B, 384] 丢进 competing-risk survival head (SurvLayer: cr)，算出痴呆和死亡两个风险的危险函数
类比
这和 NLP 里 transformer 处理"一句话"的流程完全一致——只不过：

NLP: token = word, 位置 = word index
EHR: token = medical event, 位置 = 事件发生时间（通过 delta_t 编码）
所以一个 patient = 一个"句子"，序列长度 = 这个病人在 index_date 之前有多少事件。从你的 verify 输出看 mean_len ≈ 558，说明大部分病人被 truncate 到了 512（block_size），这些病人"前面"较早的事件会被砍掉，只保留最近的 512 个。

问题 2：按 practice_id 拆分会不会丢一个病人的部分数据？
不会丢，但要理解"单位"
拆分的单位是 practice（诊所），不是 patient。流程：

practice_id_splits.pickle 里已经固定好了 {train: [practice_A, practice_C, ...], val: [practice_B, ...], test: [practice_D, ...]}
FoundationalDataModule 处理每个病人时，查他的 static_table.PRACTICE_ID，看这个 practice 落在哪个 split，这个病人就整体进那个 split
一个病人的所有事件都落到同一个 split 里，整条时间线都进去，不会被"切一半给 train 一半给 test"
为什么要按 practice 而不是按 patient 拆
这是 CPRD 类研究的标准做法，原因：

防止信息泄漏：同一个 practice 的不同病人共享医生习惯、编码风格、地区政策、设备偏差等。如果同一 practice 的病人分布在 train/test，模型可能学到 practice 层面的 shortcut 而不是疾病信号，测试集就高估了泛化能力
符合外推场景：实际部署时，模型会被部署到新的没见过的诊所，所以 test 应该模拟"来自未见 practice 的病人"
那新加进来的 HES-only 病人怎么分？
这正是 v5 流程的一个关键点：这些病人在 static_table 里本来就有 PRACTICE_ID（从 UKB 注册信息来的），所以即使他们之前没有 GP 事件也不会没人管——prepare_hes_fusion_db.py 插入 HES 事件时用的就是 static_table.PRACTICE_ID，然后 FoundationalDataModule 用同一套 practice_id_splits.pickle 自动把他们分到对应的 split。日志里那句 after insert: 466,359 patients 里多出来的 236,358 人就是这样进来的。

一个细节：序列被 truncate 不等于"拆分丢失"
即便一个病人的事件超过 512，被砍掉的是时间线最早的部分，不是被分到别的 split。信息没有跨 split 流动，只是被 transformer 的上下文窗口限制了。如果你觉得这是个瓶颈，可以考虑：

增大 block_size（代价是显存和速度）
或者改用 hierarchical / compression layer（YAML 里有 compression_layer: False 这个开关）
一句话总结：一个病人被处理成一条按 DATE 排序的 event token 序列（带 delta_t 时间编码），整条序列整体进一个 split（按 practice 划分），transformer 把它当成一个变长"句子"处理，最后一个有效 token 的 hidden state 进 survival head 出风险。你说的 "9万 × 384" 是 token embedding 矩阵的权重形状，不是单次前向传播的输入 shape。

我看了 FastEHR 的 dataloader 代码和你的 YAML 设置，发现一个非常重要的机制让你可以放心——让我完整解释。

一、"一个 event" 到底是什么
一个 event 就是 diagnosis_table（或 measurement_table、scripts、death）里的一行。具体到 diagnosis_table，一行 = (PRACTICE_ID, PATIENT_ID, EVENT, DATE)，一个 EVENT 字符串就是一个 Read v2 code。

所以"一次就诊开了 5 个诊断"就是 5 行，即 5 个独立的 token，它们共享同一个 DATE，但在序列里是 5 个连续的 token。

我刚在 DB 里实测了一下，确实见到过极端的情况——比如 patient 1000032 在 2012-05-25 这一天单日就有 43 个 diagnosis events（心血管相关的一大堆 Read 码堆在一起）:


XaIRc, XaDvl, 424.., XE2mS, 42P.., XaIdY, 42N.., XE2mO, XE2eC, X76tb, ...（共 43 个）
同一天出现"爆炸式"的事件确实存在（多半是一次彻底体检或者 HES 住院时的诊断清单）。所以你的担心是合理的。

二、截断发生在哪、怎么发生
在 FastEHR/dataloader/foundational_loader.py 第 598-653 行，流程是这样的（按顺序）：

Step A — 先去重（这一步就大幅缩短序列）
你的 YAML 里 repeating_events: False，所以在 tokenize 前有这段代码：


if not self.repeating_events:
    # 对每个 unique event 只保留"最后一次"出现的位置
    last_unique_indices = sorted({x: i for i, x in enumerate(sequence_tokens)}.values())
也就是说，一个 Read code 不管出现过多少次（同一天 43 个里有重复的、或跨年重复测高血压），只保留最新那一次。这让真实序列长度远小于原始事件数。从 verify_hes_fusion_dataset.py 的输出也能验证这一点：

中位数长度：train 228，val 395，test 176
均值长度：train 562，val 665，test 418
所以大多数病人去重后根本到不了 512。只有长尾的那一部分才会被截断。

Step B — 截断时"从尾部往回取 512"（默认 eval 模式）
第 622-627 行：


if self.random_context_window:
    start_pos = np.random.randint(...)   # 训练时可选随机窗口增广
else:
    start_pos = len(encoded_tokens) - self.max_seq_length if ... else 0
end_pos = start_pos + self.max_seq_length
默认是保留最新的 512 个 token（start_pos = len - 512），最老的历史会被砍掉。

Step C — 关键救命机制：global_diagnoses=True
你的 YAML 里有 global_diagnoses: True，这触发第 629-644 行这段代码：


if self.global_diagnoses:
    # 在被砍掉的那段前缀里，把所有属于 diagnosis_table 的事件拎出来，
    # 重新 prepend 回窗口最前面
    earlier_global_events = self.tokenizer.encode([
        _event for _event in sequence_tokens[:start_pos]
        if _event in self.meta_information["diagnosis_table"].event.tolist()

    ...
这一行是核心：即使一个 diagnosis event 位于被截掉的"远古"历史里，只要它是 diagnosis_table 里的 Read v2 code，就会被抢救回来塞到序列开头。也就是说：

被截掉的是：老的 measurements（血压、BMI 那种频繁重测的）、老的 prescriptions
不会被截掉的是：diagnosis_table 里的所有 Read v2 诊断码，包括我们刚插进去的 HES 翻译后的 dementia code、DEATH token
三、回答你最担心的问题
dementia 或重要疾病会不会被截断扔掉？

**不会。**有三重保护：

repeating_events=False 的去重让长序列的有效长度剧烈收缩，大部分病人根本到不了 512。
global_diagnoses=True 的 rescue：任何 diagnosis_table 里的 Read v2 诊断 code（包括 31 个 canonical dementia code 和 HES 翻译后的 dementia token）都会被强行塞回窗口，不管它在原始时间线里多早。
_reduce_on_outcome 先截 outcome：事实上在上面这些 slice 发生之前，index_inclusion_method 已经把序列切成"index_date 前的历史 + index_date 后的第一个 outcome"。最后一个 token 一定是 outcome 本身（k=1 的 dementia code、k=2 的 DEATH、或 k=0 的最后普通事件）。这个"最后一个 token"参与 loss 计算，永远不会被丢。
但有一个小 caveat 需要注意
global_diagnoses 救回来的是老历史里的诊断，它们被当成"不带时间上下文的背景 token"塞到窗口最前面——所以它们的准确时间顺序与相对位置会有一点点失真（它们被压扁到窗口前）。对"是否有 outcome"这种二分信号没影响，但对"什么时候发生"的细粒度时间建模略有损失。

另外，非诊断类事件（measurement、prescription）在超长序列里会被截断丢失。如果某些血压 / HbA1c 测量对 dementia 预测有价值，这部分历史会流失。但由于去重后测量的最新值都是保留的，模型仍能看到"最近的血压水平"这类信息，损失主要是老的时间序列形状。

四、验证一下你当前数据里到底多少病人被截
从 verify 输出粗估：

split	mean_len	median_len	推断
train	562.7	228	中位数远小于 512，少数病人超长
val	665.1	395	均值超 512，约一半以上被截
test	418.3	176	大多数没被截
均值 > 中位数说明是右偏长尾分布：大多数病人序列不长，但有一小撮超长病人把均值拉高。对这一小撮病人，global_diagnoses 的 rescue 机制就起作用——即便他们被截，所有 diagnosis 都会被保留下来。

一句话总结
你担心的"dementia code 被截断丢失"不会发生，因为 global_diagnoses=True 专门保护了所有 diagnosis_table 里的 Read v2 诊断码（HOTFIX 1 确保 HES dementia 翻译后就是这类码）。会被截断丢失的只是老的 measurement / prescription，它们对时间序列建模略有损失但不会影响痴呆标签识别。真正决定 k=1/k=0 的"最后一个 outcome token"由 _reduce_on_outcome 在更早的阶段就锁定，永远在序列里。
